# Peter Steinberger - "Build the thing that builds the thing" summary

Source: `Build the thing that builds the thing  BRK245 (no timeline).txt`

Note: the transcript has a few inconsistent speech-to-text names, especially around OpenCloud/OpenClaw/OpenCore and Crab/Grab/Crap Box. This summary keeps the clearest names from the transcript and focuses on the described purpose of each tool.

## High-level theme

Peter's core message is that modern software work is no longer only about writing application code faster. The higher-leverage work is building tools that help coding agents work faster, close feedback loops, verify their own work, and reduce the human's repeated friction. His recurring lesson is: when something annoys you repeatedly, treat that annoyance as a signal that a small tool or agent workflow can remove the pain.

## Key Takeaways

1. **Building the thing is never the hard thing - maintaining it is.** The real challenge isn't writing the initial code; it's the ongoing work of triage, review, releases, and keeping a growing project alive. Most of Peter's tools exist to make maintenance survivable.
2. **Laziness is good - it's where the ideas come from.** Peter treats his own unwillingness to do repetitive, annoying work as a feature, not a flaw. The friction he wants to avoid is exactly what points him toward the next tool or automation worth building.
3. **Little tools compound and make building new tools faster.** Each small helper becomes a building block for the next, so the more tools Peter builds, the faster and easier it gets to build the following ones.

## Things Peter built and talked about

### 1. OpenCloud / OpenClaw / OpenCore project ecosystem

**Description:** Peter talks about maintaining a fast-growing open-source project ecosystem, transcribed with names like OpenCloud, OpenClaw, and OpenCore. It became the main forcing function for many of his later tools.

**Pain point:** The project grew quickly, creating thousands of issues and pull requests, many of them generated by agents or low-effort prompts. Manual triage, review, releases, and maintainer coordination stopped scaling.

**How it addresses the pain point:** The project itself became the test bed for building tooling around agent-driven maintenance: automated issue triage, review, dashboards, crawlers, test boxes, visual verification, and contributor transcript collection.

**Lessons taken away:** A growing codebase needs systems around it, not just more manual effort. Issues and PRs should be treated as "prompt requests" - signals that someone wants a change - and then routed through automation wherever possible.

### 2. Cloud Sweeper / Klaus Reaper issue and PR triage

**Description:** A GitHub Action-driven tool that starts a Codex run when someone opens an issue or pull request. It reads project guidance such as `vision.md`, decides whether the request fits the project, and comments on or closes issues.

**Pain point:** Peter had more than 10,000 issues and PRs. Existing review tools only produced reviews, but did not group, close, or resolve the backlog.

**How it addresses the pain point:** The tool uses a project vision file to tell the agent what the project wants and does not want. It can close irrelevant issues, comment on useful ones, rerun weekly, detect when a main-branch fix already solved an old issue, and report activity through Discord.

**Lessons taken away:** Encode project intent so agents can apply it repeatedly. Automated triage can turn an overwhelming backlog into a manageable stream, and rerunning triage matters because the codebase changes over time.

### 3. `vision.md` and agent instruction files

**Description:** Peter uses project-level guidance files, including a `vision.md` and longer agent files, to explain goals, boundaries, invariants, and what the project does or does not want.

**Pain point:** Agents start with little memory of the project and may make technically plausible but inappropriate choices, over-defend impossible edge cases, or burn tokens on contradictory instructions.

**How it addresses the pain point:** The files tell agents what matters, what is out of scope, and which assumptions are valid. Peter also asks agents to critique and clean up their own instruction files when they become confusing.

**Lessons taken away:** The more useful project context you give agents, the better they perform. Do not rely on human memory; write down invariants, intent, and constraints. Let agents help rewrite instructions in a form that works well for agents.

### 4. Maintainer reports dashboard

**Description:** A private dashboard that reports on maintainer activity across GitHub and Discord, not just code commits.

**Pain point:** As the project gained momentum, Peter added maintainers liberally and then lost track of who was active, who was helping, and who disappeared.

**How it addresses the pain point:** The dashboard aggregates GitHub activity and Discord participation so Peter can see who is contributing through code, discussion, and maintenance work.

**Lessons taken away:** Maintenance is broader than commits. Good project health tooling should include discussion, triage, and community participation, because those are also forms of contribution.

### 5. Discord crawler / data export

**Description:** A crawler that extracts Discord data for reports and prioritization. Peter also stores serialized Discord data in GitHub so other maintainers can use it without getting the bot token.

**Pain point:** Discord data was needed for maintainer reports and project prioritization, but there was no easy public way to fetch and distribute all the useful data.

**How it addresses the pain point:** The crawler pulls Discord data, makes it usable by tools, and backs it up through GitHub as a simple shared data store for maintainers.

**Lessons taken away:** If an API or data boundary blocks a workflow, build a small bridge. The solution does not need to be perfect; it needs to be good enough to unblock automation and team access safely.

### 6. Issue browser

**Description:** A small tool that parses selected text containing issue or PR references in many formats and turns it into a fast browser for clicking through the list.

**Pain point:** Agent-generated triage output often contained many links in inconsistent formats. Manually opening each URL was too slow and annoying.

**How it addresses the pain point:** Peter can select text, feed it into the issue browser, and quickly inspect the relevant issues. Over time, he extended it to handle every odd format agents produced.

**Lessons taken away:** Optimize the moments where the human must stay in the loop. Small tools that remove "just a few clicks" compound when used repeatedly.

### 7. Gorg / Google CLI

**Description:** A CLI for accessing Google services such as Gmail and Calendar, including multiple accounts and end-to-end test support through a demo account.

**Pain point:** Google did not provide the CLI workflow Peter wanted, and existing Codex integrations were limited, including account limitations.

**How it addresses the pain point:** The CLI gives agents command-line access to Peter's Google workflows. Because Peter controls the tool, agents can add missing features and end-to-end test them against a demo account.

**Lessons taken away:** When agents can access and test the full stack, they can build, verify, and ship features with less human intervention. Owning the interface makes the system self-improving.

### 8. Prioritization crawler for "what people are screaming about"

**Description:** A workflow that connects Discord discussion, open issues, PRs, and the project vision document to find the top things the team should work on.

**Pain point:** With thousands of issues, it was hard to know which problems were actually important to users and aligned with the project direction.

**How it addresses the pain point:** The crawler lets Codex scan channels, correlate complaints with issues and PRs, match them against the vision document, and select work that agents can handle autonomously.

**Lessons taken away:** Prioritization should combine user pain, existing backlog, and project intent. Agents are most useful when they are pointed at work they can complete without constant human supervision.

### 9. Open-source project dashboard / release attention bar

**Description:** A dashboard for Peter's many open-source projects that shows what needs attention, where PRs are accumulating, and how long it has been since the last release.

**Pain point:** Across many repositories, it was hard to know which projects needed gardening, dependency updates, releases, or review attention.

**How it addresses the pain point:** The dashboard lets Peter filter projects by signals such as days since last release and focus on the repositories that need maintenance.

**Lessons taken away:** Software dies without gardening. Maintenance dashboards do not need production-grade polish if they help the maintainer make better decisions faster.

### 10. Octopus

**Description:** A GitHub API shim/proxy that uses a GitHub App token for safe public read-only operations and Peter's personal token only when necessary for mutating or unmatched operations.

**Pain point:** Agents repeatedly hit GitHub API rate limits when searching related issues and working across a large backlog. Human rate-limit resets were not reliable, especially at odd hours.

**How it addresses the pain point:** Octopus routes public read-only requests through a GitHub App, which has a larger rate limit, and falls back to Peter's token only when required. It runs through Cloudflare.

**Lessons taken away:** Platform limits become serious bottlenecks when agents scale work in parallel. Use the platform's permission and token model carefully, and review security settings, but do not accept rate limits as a permanent workflow blocker.

### 11. Crab Box / Grab Box / Crap Box

**Description:** A remote test and execution box for agents. It syncs code changes to cloud machines, runs tests on powerful hardware, supports multiple providers, and later gained cross-platform, UI, VNC, screenshot, click, and typing capabilities.

**Pain point:** Tests became slow and CPU-intensive, local machines overheated, one test provider had outages, and cross-platform bugs were painful to reproduce. UI bugs were even harder because the agent could not easily operate the environment.

**How it addresses the pain point:** Crab Box spins up cloud machines, checks out repositories, syncs changes, runs tests, falls back across providers such as AWS or Azure, and can create boxes for different operating systems or browsers. With UI automation, agents can interact with browsers and terminals directly.

**Lessons taken away:** Move expensive, repetitive verification into disposable remote environments. Parallel test boxes increase confidence, reduce local friction, and make agents more autonomous.

### 12. Mantis

**Description:** A visual verification agent that can be pinged on a pull request. It spins up boxes, records a video of the bug, applies or verifies a fix, records a video of the fixed behavior, and checks the result.

**Pain point:** Even with remote UI boxes, Peter still had to manually watch or click through scenarios to confirm that bugs were reproduced and fixed.

**How it addresses the pain point:** Mantis gives the agent a visual loop: reproduce the bug, make a video, fix the issue, make a video of the fix, and verify the visual result. Peter only needs to review the evidence and merge.

**Lessons taken away:** The most powerful automation closes the feedback loop, not just the coding loop. Agents need ways to see, act, and verify outcomes like humans do.

### 13. Auto Review

**Description:** An automated multi-round review workflow triggered by agent instructions before commit or PR landing. Codex can call another Codex instance with a fresh context to review the work.

**Pain point:** Manual `/review` loops were slow and repetitive. Each review found a few issues, the agent fixed them, then another review found more, sometimes consuming an entire day.

**How it addresses the pain point:** Peter added an instruction to his agent file: before committing or landing a PR, run auto review if it has not already been done. The workflow runs multiple review rounds and includes project-specific rules about which findings are valid.

**Lessons taken away:** Fresh-context review improves trust in agent-generated code. Automate the review loop, but teach the reviewer agent the project's real invariants so it does not waste time on irrelevant edge cases.

### 14. Contributor prompt/transcript upload skill

**Description:** A contribution workflow skill for OpenClaw/OpenCloud contributors that asks whether they want to upload a privacy-filtered version of the prompt transcript to the PR.

**Pain point:** Peter wanted to know how a PR was produced and how much effort went into it, but contributors did not manually include prompts even when guidelines asked them to.

**How it addresses the pain point:** The agent asks the contributor for permission, finds its own transcript on disk, trims it to the relevant parts, privacy-filters it, and attaches it to the pull request.

**Lessons taken away:** Do not fight user laziness with policy alone. Build the desired behavior into the workflow, reduce the contributor's effort, and use prompt history as a signal for PR quality.

### 15. Cloud Patch

**Description:** A large-codebase review tool that splits a big project into many smaller subsections and has agents review each area separately.

**Pain point:** Peter felt uneasy about old code that had never gone through agent review, but a million-line project is too large for a single slash-review context.

**How it addresses the pain point:** Cloud Patch decomposes the project into smaller feature or subsystem areas, reviews them independently, and surfaces cleanup opportunities.

**Lessons taken away:** Large codebases need decomposition before agents can reason about them well. If the whole project does not fit the context, create smaller review units.

### 16. Crab Fleet / Crap Fleet

**Description:** A multiplayer Codex workflow built on top of Crab Box. It lets collaborators open a shared session, talk to the agent, observe it, or take over.

**Pain point:** When teammates collaborated, one person often acted as an intermediary, copying messages between team chat and the coding agent. Peter wanted to talk directly to the teammate's agent.

**How it addresses the pain point:** Crab Fleet runs Codex inside the shared box environment and exposes sessions through links so maintainers can inspect, converse with, and control the same agent session.

**Lessons taken away:** Agent collaboration should be direct, not relayed through humans. Tools compound: once Crab Box existed, multiplayer agent work became much easier to build.

### 17. Click Clock

**Description:** A messaging platform built for controlled end-to-end tests of model-message channels and complex messaging features. It also became a backup chat system.

**Pain point:** Testing against external messaging platforms was difficult because automation triggered captchas, Cloudflare checks, and other anti-bot protections. Some API behavior was easier to understand through a UI.

**How it addresses the pain point:** Peter's team built their own controlled messaging platform so agents could run realistic end-to-end tests without being blocked by third-party anti-automation systems.

**Lessons taken away:** When external systems make automation unreliable, build a controlled test environment. A tool built for testing can become operationally useful in unexpected ways.

## Cross-cutting lessons

1. **Annoyance is a signal.** Repeated irritation usually means there is a loop worth automating.
2. **Close the loop for agents.** Give agents screenshots, browsers, terminals, test boxes, videos, and APIs so they can verify their own work.
3. **Build small, sloppy internal tools.** Helper tools do not need production polish if they reduce friction and are not system-critical.
4. **Encode intent.** Project vision, invariants, and agent instructions let agents make better decisions without asking humans every time.
5. **Prioritize human attention.** Keep humans in the loop for judgment, risky decisions, and ambiguous work; automate the rest.
6. **Let tools compound.** Each small tool makes the next one easier to build, and agents can reuse patterns from previous tools.
7. **Review and verification build trust.** Auto review, cross-platform testing, and visual verification make agent-generated changes safer to merge.
8. **Do not wait for perfect products.** If a commercial or platform tool does not exist, or does not quite fit, build the rough version that removes your specific pain.
